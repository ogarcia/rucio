# Web UI

## Overview

`rucio-web` is a Leptos 0.8 client-side rendering (CSR) application compiled
to WASM by [trunk](https://trunkrs.dev/).  The build output is a small set of
static files (`index.html`, a `.js` loader and a `.wasm` bundle) that work
in two deployment modes:

| Mode | How it works |
|---|---|
| **Embedded** | Built into the daemon binary via `rust-embed`; served by Axum at `/`. Enable with `--features web-ui`. |
| **Standalone** | Copy `rucio-web/dist/` to any static file server (nginx, Caddy, Apache). The UI calls the daemon's REST API directly. |

---

## Prerequisites

- `wasm32-unknown-unknown` Rust target
- `trunk`

See [01-toolchain.md](01-toolchain.md) for installation instructions.

## Development workflow

trunk's dev server compiles on save and hot-reloads the browser.  Start it
from inside `rucio-web/`:

```sh
cd rucio-web
trunk serve --proxy-backend=http://127.0.0.1:7070/api
```

`--proxy-backend` forwards `/api/*` requests to a running daemon so the UI
can reach real data without CORS issues.  The daemon must be running
separately:

```sh
# In another terminal, from the workspace root
cargo run --bin ruciod
```

trunk serves the frontend at `http://127.0.0.1:8080` by default.

## Building for production

```sh
cd rucio-web
trunk build --release
```

Output goes to `rucio-web/dist/`.  The directory is gitignored — it contains
only generated artefacts.

### Embedded in the daemon

From the workspace root, after `trunk build --release`:

```sh
cargo build --release -p rucio-daemon --features web-ui
# or the fat binary:
cargo build --release -p rucio --features web-ui
```

`rust-embed` reads `rucio-web/dist/` at compile time and bakes the assets
into the binary.  The daemon serves them via Axum's `fallback` handler:
API routes at `/api/*` take precedence; every other path falls through to
the SPA.

### Standalone (nginx example)

```nginx
server {
    listen 80;
    root /srv/rucio-web;
    index index.html;

    # Proxy API calls to the daemon
    location /api/ {
        proxy_pass http://127.0.0.1:7070;
    }

    # SPA fallback — unknown paths return index.html
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

Copy `rucio-web/dist/` to `/srv/rucio-web` and reload nginx.

## Architecture notes

### Why `spawn_local` instead of Leptos `Resource`

`Resource::new` in Leptos 0.8 encodes the resource value as JSON for
server-side rendering hydration.  This imposes two constraints that conflict
with a pure CSR WASM app:

1. The output type must implement `serde::Serialize`.
2. The future must be `Send`.

`gloo-net`'s HTTP futures are not `Send` (they wrap `JsFuture`, which holds
an `Rc`).  Since there is no SSR in this application, both constraints are
unnecessary.  The UI uses `RwSignal<Option<Result<T, E>>>` + `spawn_local`
instead — the signal holds the latest fetch result and the component reads
it reactively.

### File layout

```
rucio-web/
  Cargo.toml        # Leptos + gloo-net + serde; compiles to wasm32
  index.html        # trunk entry point; contains all CSS inline
  src/
    main.rs         # App component, StatusCard component, fetch logic
  dist/             # Generated by trunk — gitignored
```

### Adding new views

The UI calls `GET /api/v1/status` on load and on each "Refresh" click.
To add a new view (e.g. downloads list):

1. Add a `fetch_downloads()` async function that calls `GET /api/v1/downloads`.
2. Add a `DownloadsCard` component that receives the response as a prop.
3. Add a `RwSignal` for the downloads state in `App` and call `spawn_local`
   to populate it alongside the status fetch.
